使用nmap工具对IP地址192.168.2.88进行全端口扫描,即扫描该IP地址从1-65535的所有TCP端口。

通过这条命令可以检测目标主机是否开放了全部TCP端口,获取目标主机打开的TCP服务类型。

如果端口打开,还可以使用其他nmap参数如-sV进行版本检测。

它是一条用于全面探测目标主机开放端口状况的基础nmap命令。通过全端口扫描可以得到目标主机端口开放的完整情况。

使用nmap进行ping扫描,检测192.168.2.0-192.168.2.255网段内主机是否存活,但不进行端口扫描。

通过-sn参数只发出ICMP回显请求包(ping包),快速检测网络段内主机是否可达,但不会检测主机开放的端口。

这主要用于初步探测网络拓扑结构,快速获取网络段内活跃主机的IP列表。

使用nmap对IP地址192.168.2.88进行端口扫描,但只扫描目标从1至500的TCP端口号。

通过限定端口范围-p1-500,可以缩小扫描范围,避免全端口扫描需要的长时间。

这主要用于快速扫描常用端口,获取目标主机可能开放的服务类型,如HTTP、FTP等。

CDN(内容分发网络)

CDN会储存服务器发送的副本，用户可以通过就近的CDN来访问服务器内容，如果服务器有新的内容，就会将新内容发送给CDN。

内容分发网络的英文缩写(Content Delivery Network)。它是一个分布式的系统,将Web内容存储到世界各地的边缘位置,从而内容可以离用户更近,访问速度更快。

CDN作用：

  • 提高访问速度:通过将内容缓存到边缘节点,大大缩短用户访问内容的距离,提高访问速度。

  • 分担源站压力:将部分流量引导到边缘节点处理,减轻源站服务器的负载。

  • 提高可用性:通过分布式节点,一旦部分节点宕机,不影响整体访问。

  • 安全防护:CDN节点可以过滤攻击流量,为源站提供一定程度的保护。

  • 负载均衡:CDN可以根据用户地理位置自动选择就近节点,实现全球负载均衡。

dig命令用于查询域名系统(DNS)信息。

命令dig qq.com会返回qq.com域名的以下DNS信息

  • QUESTION SECTION查询的域名
  • ANSWER SECTION返回的qq.com的IPv4地址
  • ADDITIONAL SECTION返回的qq.com的IPv6地址
  • SERVER查询使用的DNS服务器IP
  • WHEN查询时间
  • MSG SIZE返回信息包大小

dig命令查询域名DNS记录,用于测试域名解析是否正常。

nslookup命令用于查询域名系统信息。

命令nslookup qq.com会返回qq.com域名的DNS查询结果

  • Server: 查询使用的DNS服务器IP地址
  • Name: 查询的域名
  • Address: 域名对应的IPV4地址

与dig命令不同的是,nslookup返回的结果直接显示域名对应的IP地址,而不显示额外的DNS记录类型等详细信息。

此外,nslookup返回结果不是权威的,因为它使用公共DNS服务器查询,而不是从域名注册机构获取权威响应。

总的来说,nslookup命令用于快速查询域名到IP的映射,验证域名是否可以正常解析;dig命令返回更全面详细的DNS记录信息,用于调试和分析域名系统设置。

业务逻辑漏洞的成因

  • 应用程序在设计和实施的时候存在缺陷,允许攻击者诱发意外行为,像是设计的方案被人钻了空子。 通俗理解就是:在编写程序时,只考虑了常规的操作流程,即“当在A情况下，就会出现B，此时执行C即可”，但是开发者却没有考虑当用户执行了意料之外的操作时会发生什么。这种对于异常情况的欠考虑，最终导致了安全漏洞的产生。

  • 常见的OWASP漏洞，通过漏洞扫描工具，大多支持自动化或者半自动化扫描出来；并且传统的安全防御设备和措施收效甚微；

  • 但逻辑漏洞属于和系统自身功能和逻辑有关系的漏洞，每一家的漏洞出现可能存在一定的独特性，很难复制或者通过规则通过脚本扫描，因此逻辑漏洞大多需要配合代码审计和手动测试才可发现相关漏洞，也是工具无法完全替代人所作的一类漏洞。

  • 业务逻辑属于正常请求，都是按照正常的程序走的，一般的安全设备是无法拦截的。

分类：

  1. 验证码突破
  2. 业务授权安全
  3. 业务流程乱序
  4. 业务接口调用
  5. 时效绕过测试
  6. 身份认证安全
  7. 业务一致性安全
  8. 业务数据篡改
  9. 用户输入合规性
  10. 密码找回漏洞

支付逻辑漏洞是指系统的支付流程中存在业务逻辑层面的漏洞

常见支付流程：
选择商品和数量——选择支付及配送方式——生成订单编号——订单支付选择——完成支付

如：最常见的支付逻辑漏洞通常是由于服务器端没有对客户端请求数据中的金额、数量等敏感信息进行效验导致。一般漏洞产生在电子商务类应用中。

支付逻辑漏洞一般分为四类：

1.支付过程中可以修改支付金额 2.可以将订单中的商品数量修改为负值 3.请求重放导致 4.其他问题（程序异常，其他参数修改导致问题等） 其他支付问题补充： 修改支付状态、修改附属值（优惠券积分等）、修改支付接口、替换多个订单、重复支付等。

支付逻辑漏洞之修改订单数量：
1、登录网站，选择购买一个商品并抓取数据包

2、找到其中代表商品数量的参数，将参数的值修改为负数

弱口令

网站管理、运营人员由于安全意识不足，为了方便、避免忘记密码等，使用了非常容易记住的密码，或者是直接采用了系统的默认密码等。

攻击者利用此漏洞可直接进入应用系统或者管理系统，从而进行系统、网页、数据的篡改与删除，非法获取系统、用户的数据，甚至可能导致服务器沦陷。

弱口令(weak password) 没有严格和准确的定义，通常认为容易被别人（他们有可能对你很了解）猜测到或被破解工具破解的口令均为弱口令。弱口令指的是仅包含简单数字和字母的口令，例如“123456”、“abc”等，因为这样的口令很容易被别人破解，从而使用户的计算机面临风险，因此不推荐用户使用。

简单密码

过于简单直白的口令,如123456、12345678等数字和字母组合。

与账户相关的易知信息组成,如用户名、email后缀作为密码等。

词典词汇,如名字、地点名、简单词汇组成的密码。

系统默认口令未及时修改,如生产设备默认口令等。

口令长度过短,通常小于8位口令较易破解。

口令使用过于单一简单的模式,如只使用字母或只使用数字等。

口令爆破

口令爆破其实就是攻击者对用户口令进行穷举尝试。在用户登录界面，攻击者通过遍历生成口令或者加载口令字典，进行多次登录尝试，直至把用户口令穷举出来，成功登录

因为一般能被成功猜测出的用户口令都是弱口令，所以口令爆破也称弱口令漏洞

口令破解的漏洞原理其实很简单，即网站未对用户登录认证的请求频率和次数进行限制，使得攻击者可以无数次地进行遍历口令登录。从理论上讲，只要加载的字典范围足够大，尝试的次数足够多，攻击者可以把任意用户的口令穷举出来

相对-安全口令

1.不使用空口令或系统缺省的口令，因为这些口令众所周知，为典型的弱口令。

2.口令长度不小于8个字符。

3.口令不应该为连续的某个字符（例如：AAAAAAAA）或重复某些字符的组合（例如：tzf.tzf.）。

4.口令应该为以下四类字符的组合，大写字母(A-Z)、小写字母(a-z)、数字(0-9)和特殊字符。每类字符至少包含一个。如果某类字符只包含一个，那么该字符不应为首字符或尾字符。

5.口令中不应包含本人、父母、子女和配偶的姓名和出生日期、纪念日期、登录名、E-mail地址等等与本人有关的信息，以及字典中的单词。

6.口令不应该为用数字或符号代替某些字母的单词。

7.口令应该易记且可以快速输入，防止他人从你身后很容易看到你的输入。

8.至少90天内更换一次口令，防止未被发现的入侵者继续使用该口令。

因此我们在此想办法将 ‘ 前面添加的 \ 除掉，一般有两种思路：
1、%df 吃掉 \ 具体的原因是 urlencode(‘) = %5c%27，我们在%5c%27 前面添加%df，形
成%df%5c%27，而上面提到的mysql在GBK编码方式的时候会将两个字节当做一个汉字，此事%df%5c就是一个汉字，%27则作为一个单独的符号在外面，同时也就达到了我们的目的。
2、将'中的\过滤掉，例如可以构造%**%5c%5c%27的情况，后面的%5c会被前面的%5c
给注释掉。这也是bypass的一种方法。

工作模式：

Apache一共有3种稳定的MPM模式(MPM：多进程处理模块)，它们分别是 prefork、worker、event；

prefork 工作模式

Apache在启动之初， 就预先fork一些子进程， 然后等待请求进来。之所以这样做， 是为了减少频繁创建和销毁进程的开销。每个子进程只有一个线程， 在一个时间点内，只能处理一个请求。

  • 优点： 成熟稳定，兼容所有新老模块。同时， 不需要担心线程安全的问题。
  • 缺点： 一个进程相对占用更多的系统资源，消耗更多的内存。而且，它并不擅长处理高并发请求。

通俗解释

用更通俗直观的语言来解释Apache网页服务器工作原理:

Apache服务器就像一个24小时值守的门卫。它会一直监视着网站的大门,等待访客上门。

当有人来访时,门卫Apache会第一时间走出来迎接。它会问访客一些问题,比如姓名,来访目的等。这就是解析请求的过程。

门卫Apache会根据访客的要求,寻找网站内相关的文件或服务。就像帮助访客找到正确的人或部门。如果文件在网站内,它就会帮助访客取得文件。

门卫Apache还会检查访客的身份,是否有权限访问这些文件和服务。如果没有权限,就不允许访问。这是权限检查的过程。

取得文件后,门卫Apache会告诉访客一些基本信息,比如文件类型是什么。然后交给访客传阅。这是响应的过程。

整个访问结束后,门卫Apache会礼貌送访,并记录此次访问情况,以便日后参考。它还会及时应对更多访客,确保网站高效运行。

所以,Apache就像一个24小时值守网站的门卫,它会礼貌高效地为每一位访客提供服务,保证网站的正常访问。

ascii => 编排了128个文字符号。只需要7个0和1就可以表示了。

中国：gb2312编码，gbk编码 (windows，默认编码),

国家之间编码方式不同，一样的数字有不一样的文字

Unicode: 万国码。

文字量太大，从UCS-2 2个字节，扩充到了UCS-4 4个字节

utf:是可变长度的unicode。可以进行数据的传输和存储。

utf-8: 最短的字节长度8

​ 英文： 8bit，1byte

​ 欧洲文字：16bit，2byte

​ 中文：24bit，3byte

utf-16：最短的字节长度16

总结：

  1. ascii:8bit,1byte

  2. gbk:16bit,2byte windows默认

  3. unicode:32bit,4byte(没法用，只是一个标准)

  4. utf-8: mac默认

    ​ 英文： 8bit，1byte

    ​ 欧洲：16bit，2byte

    ​ 中文：24bit，3byte

gbk和utf-8不能直接就进行转化

gbk <=> 文字 <=> utf-8
